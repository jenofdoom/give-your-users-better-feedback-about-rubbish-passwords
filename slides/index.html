Give your users better feedback about rubbish passwords with zxcvbn

Jen Zajac // @jenofdoom

Catalyst

Passwords are fundamentally broken

Most common passwords

  1. 123456
  2. password
  3. 12345
  4. 12345678
  5. football
Source: SplashData, 2016

As devs or infosec folks, we know better

we use password managers, so we don’t reuse the same passwords

we follow xkcd’s advice about correct horse battery staplers

we always enable 2FA, maybe even using a physical 2FA token

Our nontechnical users… often don’t know this stuff yet

In the absence of external guidance, we can’t blame our users for this

(nor would blame help fix anything)

The list of most common passwords hasn't changed very much over the years - just telling people that they're messing this up isn't helping. If we want things to be more secure we need to enforce standards ourselves.

In fact, as an industry, we’ve systematically created a set of perverse incentives that make it more likely that users will create bad passwords

username + password is an overused hammer in a sparse toolbox

Without the crutch of a password manager, is it any wonder that in the modern world we live in, users reuse passwords?

we provide bad advice about what will make a password more secure

Using a capital letter and a number in your dictionary word no longer cuts the mustard. Most password strength validation relies on counts of lower and uppercase letters, digits and symbols.

we forbid arbitrary characters

Many sites even prevent the use of symbols at all.

our minimum length requirements are often outdated

With modern password cracking hardware, 6 character or less passwords are very insecure.

Overall, we fail to provide good advice

Fortunately, we can do better

zxcvbn

zxcvbn has been around since 2012, created by an engineer at DropBox | open source library, written in JavaScript (well, actually CoffeeScript) | there are many ports to various other languages, and it can also be used as b/e JS

Demo

Today we'll be focusing on how to integrate it with your frontend. In practice you'd definitely want it on both your frontend and backend so you don't have rules mismatches.
